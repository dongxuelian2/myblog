# Design System

## 核心目标

这套博客设计系统只服务五个词：优雅、流畅、生动、简洁、准确。

它不是“给每个页面各写一套好看的样式”，而是把页面结构、设计令牌、组件版型、图片策略和排版规则明确写死，让后续新增页面时仍然维持同一种气质。

参考方向来自以下页面在 2026 年 3 月 19 日可见的公开结构特征：

- `https://claude.com/blog`
- `https://claude.com/blog/organization-skills-and-directory`
- `https://openai.com/zh-Hans-CN/index/small-business-stories/#a-seed-farm-in-south-carolina`

提炼出的共同点：

- 页面底色偏暖白，强调“纸面感”，而不是高饱和产品营销感。
- 标题具备编辑式张力，但按钮和控件非常克制。
- 图片承担节奏切换和叙事作用，不抢正文。
- 卡片有统一骨架，边界清楚，动效非常轻。
- 长文以阅读为中心，不让旁支组件破坏视线。

## 文件职责

- `design.md`
  设计规范唯一来源。决定页面必须包含哪些元素、颜色和字体如何使用、图片比例如何控制、间距如何统一。
- `tailwind.config.ts`
  设计令牌实现层。颜色、阴影、字体、宽度、排版主题都在这里集中定义。
- `assets/css/main.css`
  全局基础样式与组件层。只放基础 reset、站点级公用类、排版类，不再堆叠零散页面 CSS。
- `components/site/*`
  站点骨架组件。只处理导航、页脚、整体框架。
- `components/ui/*`
  页面级通用模块。比如统一的 Page Hero。
- `components/posts/*`
  文章域组件。只负责文章列表、筛选、搜索、卡片和文章相关 UI。

## 设计令牌

### 色彩

主色调采取“暖白纸面 + 深墨文字 + 节制强调色”。

| 角色             | 色值      | 用途                           |
| ---------------- | --------- | ------------------------------ |
| `paper`          | `#f6f1e8` | 全站背景，提供纸面感           |
| `surface`        | `#fffdf8` | 卡片、封面容器、正文背景       |
| `surface-muted`  | `#efe7db` | 次级分组、占位图底色、引用背景 |
| `ink`            | `#161412` | 主标题、关键文案               |
| `ink-soft`       | `#3f3a34` | 正文与说明文字                 |
| `ink-subtle`     | `#726b63` | 标签、辅助信息、时间           |
| `outline`        | `#d9cfbf` | 常规描边                       |
| `outline-strong` | `#b8ad9d` | hover / 激活边界               |
| `sunrise`        | `#d18459` | 唯一主强调色                   |
| `sea`            | `#8da5b3` | 图片与信息型点缀               |
| `moss`           | `#89986f` | 次级自然色点缀                 |

约束：

- 页面中长期活跃的强调色只能有一个主角色，默认为 `sunrise`。
- `sea` 和 `moss` 只能用于插图、封面占位与少量辅助气氛。
- 标签、按钮、边框和卡片不允许大面积彩色化。

### 字体

- UI / 导航 / 按钮 / 正文默认：`Noto Sans SC`
- 标题 / 引言 / 引用：`Noto Serif SC`
- 代码：`JetBrains Mono`
- 长文正文沿用 `Noto Sans SC` 这一组阅读字体，避免引入额外的大体积字体资源

约束：

- 中文标题不额外拉大字距。
- 权重只使用 `400 / 500 / 600` 三档。
- Serif 只用于标题、长引言、blockquote，不大面积铺满 UI。

### 宽度

- `shell`: `80rem`
  全站最大外层容器
- `layout`: `72rem`
  列表页、关于页、详情页头部主容器
- `reading`: `46rem`
  正文阅读列宽

约束：

- 任何正文都不能直接铺满 `layout` 宽度。
- 长文阅读列固定向 `reading` 收束，保证每行汉字数量稳定。

### 阴影与圆角

- 卡片基础圆角：`2rem`
- 小型胶囊：全圆
- 基础阴影：`shadow-soft`
- 强化阴影：`shadow-panel`

约束：

- 不使用巨大浮夸投影。
- hover 只允许轻微抬升和边框增强，不做夸张放大。

## 页面蓝图

### 首页

必须包含：

1. 统一 Page Hero
2. 一张精选文章卡片
3. 一组站点原则说明
4. 最近更新网格

不允许：

- 首页承担复杂筛选逻辑
- 首页同时出现多套不同样式的文章卡片
- 首页堆满介绍性废话而没有内容入口

### 文章归档页

必须包含：

1. Page Hero
2. 搜索输入框
3. 标签 chips
4. 排序控件
5. 一张精选卡
6. 统一标准卡片网格

约束：

- 搜索始终可见，但只占一块安静的输入区
- 标签数量控制在有限范围，不做多层筛选器
- 文章卡片之间使用统一高度关系和图片区比例

### 文章详情页

必须包含：

1. 面包屑返回
2. 标题区
3. 副标题或摘要
4. 日期 / 阅读时间 / 标签
5. 可选封面
6. 正文
7. 上下篇导航
8. 相关文章

不允许：

- 正文开头塞入复杂作者卡
- 正文左右堆满漂浮按钮
- 正文列宽失控

### 关于页

必须包含：

1. 站点定位 Hero
2. 设计原则卡片
3. 页面蓝图说明
4. 组件规则摘要

它的任务是解释系统，不是再做一个“伪首页”。

## 组件规范

### Header

- 高度约 `64px`
- 半透明暖白背景 + blur
- 只放品牌、主导航、一个轻量入口链接
- 导航项使用胶囊激活态，不做重色块按钮

### Page Hero

- 结构固定为：eyebrow + title + description + meta pills + optional actions
- 标题最大宽度不超过 `4xl`
- 说明文字最大宽度不超过 `2xl`
- Hero 结束后必须以细分割线收尾

### 按钮

只保留三类：

- Primary
  深色实底，少量核心动作使用
- Secondary
  白底细边框，次级操作使用
- Text Link
  文字链接，保留下划线或细线反馈

### 标签 Chips

- 高度约 `38px`
- 中性背景
- 激活态通过边框、文字和轻微阴影增强
- 不使用彩色分类墙

### 文章卡片

只允许两种：

- Featured
  横向或分栏布局，承担页面锚点
- Standard
  统一网格卡片，顶部媒体区 + 正文区

共同规则：

- 标题必须左对齐
- 摘要限制在 2 到 3 行阅读密度
- 元信息全部使用统一 `meta-pill`
- hover 只轻微上浮

### 图片

图片不再“有就随便塞，没有就完全不管”，而是统一纳入版式。

规则：

- 首页与归档页卡片图片：`4:3`
- 详情页封面：`16:9`
- 图片必须 `object-cover`
- 圆角与卡片系统保持一致
- 没有真实图片时，使用抽象封面 SVG 或统一占位图，而不是留下空白洞

说明：

- 抽象封面负责维持页面节奏与视觉秩序
- 真图加入后也必须遵守相同比例和裁切规则

## 间距体系

只使用清晰的节奏层级：

- `12px`
  小型文字与控件内部细间距
- `16px`
  常规组件内部元素间距
- `20px`
  卡片内容组间距
- `24px`
  卡片内主要区块间距
- `40px`
  页面模块间距
- `64px`
  页面大段落与区块之间的主节奏

约束：

- 同一页面不要同时出现随机的 `18 / 26 / 34 / 52px` 这类游离值
- 区块间距大于组件间距，大标题与正文之间的距离必须稳定

## 正文排版

正文策略以中文阅读为第一优先级：

- 字号：桌面 `17px` 左右
- 行高：`1.9`
- 阅读列宽：`46rem`
- 段落之间使用稳定的纵向节奏
- 标题使用 serif，正文使用 sans / reading font
- 引用块使用轻背景、细左边线，不做厚重色块
- 行内代码必须有轻微底色与边框
- 代码块必须能横向滚动，不能挤爆布局

## 交互与动效

- 动效时长控制在 `160ms - 220ms`
- 只允许位移、边框和阴影的轻变化
- 页面不使用强弹簧感、夸张缩放和旋转

目标不是制造存在感，而是让交互“顺”。

## 新增页面或组件时的检查清单

1. 页面是否属于现有四种蓝图之一
2. 如果不是，是否已经先更新 `design.md`
3. 是否复用了已有 Hero、按钮、卡片、标签和输入组件
4. 是否遵守统一的容器宽度和图片比例
5. 是否让正文阅读保持在 `reading` 列宽
6. 是否避免给单页发明一套新色板或新圆角

如果做不到以上六点，就不要直接写样式，先回到设计规则层修正。
